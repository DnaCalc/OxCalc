§01 · Executive overview

What OxCalc is — and is not

OxCalc owns the core calculation engine for DNA Calc. Its responsibilities are deliberately narrow and load-bearing:

  1. Hold the immutable structural truth of a tree-shaped calc graph (nodes, parents, projections, formula handles).
  2. Lower formulas to a dependency descriptor set, build the dependency graph, and detect cycles.
  3. Drive the recalc state machine: dirty → needed → evaluating → publish-ready (or rejected).
  4. Govern the capability-fenced overlay store that protects in-flight execution state from premature eviction.
  5. Run a single-publisher coordinator that admits, accepts, rejects, and publishes candidate work atomically.
  6. Hand evaluation to OxFml through a minimal upstream-host packet — and validate every commit bundle on the way back.

What OxCalc explicitly is not

  • It is not a host UI, document loader, file-format adapter, or workbook editor.
  • It does not own formula syntax, parsing, binding, or function-library semantics — that is OxFml & OxFunc.
  • It does not publish without coordinator authority; no evaluator may smuggle results around the seam.
“Structural truth is immutable. Identity precedes projection. Runtime state is derived and versioned.”
— CORE_ENGINE_STATE_AND_SNAPSHOTS.md, the kernel’s constitution.

§02 · Theoretical foundation

The six state-kernel principles

Six rules anchor everything else. They are constraints first, design choices second — they define what OxCalc may and may not do with state.

Rule 1

Structural truth is immutable

Any persistent change produces a new snapshot; existing snapshots never mutate. Snapshots are reference-counted, value-typed, and distinguishable by StructuralSnapshotId.

Implementation: StructuralSnapshot::create validates and freezes; apply_edit returns a successor.

Rule 2

Identity precedes projection

TreeNodeId(u64) is engine truth; the slash-separated projection path "Root/Branch/Leaf" is derived. Renames invalidate paths but never identity.

Rule 3

Runtime state is derived and versioned

Invalidation, overlays, publications, and caches are runtime state, attached to a structural snapshot via explicit fence keys (structural_snapshot_id + compatibility_basis).

Rule 4

Pinned readers see stable state

A pinned reader binds a snapshot id, a publication id (or none), and a values map. The view is value-frozen even as the live engine moves on.

Rule 5

No structural smuggling through runtime mutation

If a fact is structural, it must live in the snapshot. Caches may be lossy; structure may not. This rule kills a whole class of correctness drift before it starts.

Rule 6

Future concurrency must not force a redesign

The kernel is already shaped for staged concurrency. Stage 1 (sequential) produces the same observable artifacts that Stage 2 (partitioned) and Stage 3 (advanced lanes) will publish.

Why this matters in code

These six rules collapse into a few practical patterns you will see everywhere in the OxCalc Rust source:

  • Most public types are Clone, PartialEq, and Eq. Determinism is enforced by structure.
  • Mutating containers are BTreeMap / BTreeSet — sorted iteration order is part of the contract.
  • Edits return outcomes bundled with an impact classification (rebind required, recalc only, removal, no-rebind), not just a new state.
  • Every overlay key carries the snapshot id and a compatibility basis — overlays from a stale snapshot cannot be silently reused.
  • The coordinator never publishes without an explicit accepted candidate; MissingAcceptedCandidate is a real error.

The compact data model

Two universes, one bridge: every overlay and publication record knows which snapshot it belongs to.

§03 · Anatomy of structural truth

The StructuralSnapshot

The snapshot is the only place a node’s parent, children, symbol, formula handle, or constant value may live as truth. Every read goes through it; every edit produces a successor of it.

Core types

// src/oxcalc-core/src/structural.rs
pub struct TreeNodeId(pub u64);
pub struct StructuralSnapshotId(pub u64);
pub struct FormulaArtifactId(pub String);
pub struct BindArtifactId(pub String);

pub enum StructuralNodeKind {
    Root, Container, Calculation, Constant,
}

pub struct StructuralNode {
    pub node_id:             TreeNodeId,
    pub kind:                StructuralNodeKind,
    pub symbol:              String,
    pub parent_id:           Option<TreeNodeId>,
    pub child_ids:           Vec<TreeNodeId>,
    pub formula_artifact_id: Option<FormulaArtifactId>,
    pub bind_artifact_id:    Option<BindArtifactId>,
    pub constant_value:      Option<String>,
}

pub struct StructuralSnapshot {
    snapshot_id:   StructuralSnapshotId,
    root_node_id:  TreeNodeId,
    nodes:         BTreeMap<TreeNodeId, StructuralNode>,
    path_index:    BTreeMap<String, TreeNodeId>, // projection
}

All fields private; construction goes through create() which validates the tree and builds the projection-path index in one pass.

Construction guarantees

StructuralSnapshot::create rejects:

  • MissingRoot · declared root not present
  • RootHasParent · root with a parent_id
  • CycleDetected · a node revisited during DFS
  • MissingChild · child id not in the map
  • ParentMismatch · child’s parent_id ≠ owning parent
  • DetachedNodes · nodes unreachable from root
  • DuplicateProjectionPath · path collision in index
Why a path index? Resolving "Root/Branch/Leaf" to a TreeNodeId in O(log n) lets formulas use path-shaped references without rewalking the tree on every lookup.

Example: a 4-node tree

Node identity is a u64. The projection path is computed from the chain of symbols.

Query API (selected)

§04 · Edits and impact

Structural edits & impact taxonomy

Edits never mutate. apply_edit takes the current snapshot, the requested edit, and a successor id, and returns a StructuralEditOutcome that says exactly what kind of downstream pressure the edit creates.

The four impact classes

NoRebind Pure cache invalidation. No reference can have moved.
RecalcOnly Values changed; binding is still valid. Dependent nodes only need recalc.
RebindRequired Symbol moved/renamed; bindings that resolved through path or relative shape may now resolve differently.
Removal Subtree gone. Any descriptor pointing into it must be re-checked.

Edit menu

  • RenameNode — touches symbol
  • MoveNode — re-parents a subtree
  • ReplaceFormulaAttachment — swaps formula/bind handles
  • SetConstantValue — changes a constant payload
  • InsertNode — adds a node under a parent
  • RemoveNode — drops a subtree (root excluded)
Why classify impact? The impact classification is consumed by derive_structural_context_invalidation_seeds to decide whether dependents need StructuralRebindRequired or merely StructuralRecalcOnly seeds. Wrong classification ⇒ wrong invalidation ⇒ silent staleness.

Edit pipeline

What a real outcome looks like

// from structural.rs unit test
let moved = snapshot.apply_edit(
    StructuralSnapshotId(2),
    StructuralEdit::MoveNode {
        node_id: TreeNodeId(4),
        new_parent_id: TreeNodeId(3),
        new_index: None,
    },
).unwrap();

assert_eq!(moved.impact, StructuralEditImpact::RebindRequired);
assert_eq!(
    moved.snapshot.get_projection_path(TreeNodeId(4)).unwrap(),
    "Root/Sibling/Leaf");  // path moved with the node

Edit→seed plumbing

Downstream, derive_structural_context_invalidation_seeds walks edit_outcomes, computes whether rebind pressure is present (RebindRequired or Removal), and emits one seed per formula owner with the correct InvalidationReasonKind:

if outcome.impact == RebindRequired ||
   outcome.impact == Removal {
    rebind_pressure_present = true;
}
// per owner: requires_rebind ⇒ StructuralRebindRequired
//             else            ⇒ StructuralRecalcOnly

§05 · Reference algebra

Ten shapes of TreeReference

A formula’s power is the set of ways it can name another node. OxCalc enumerates ten reference shapes, each with explicit resolution rules and an explicit policy on whether a structural edit forces a rebind.

pub enum TreeReference {
    DirectNode          { target_node_id: TreeNodeId },
    ProjectionPath      { projection_path: String },
    RelativePath        { base, path_segments },
    SiblingOffset       { offset: isize, tail_segments },
    HostSensitive       { carrier_id, detail },
    CapabilitySensitive { carrier_id, detail },
    ShapeTopology       { carrier_id, detail },
    DynamicPotential    { carrier_id, detail },
    DynamicResolved     { target_node_id, carrier_id, detail },
    Unresolved          { token: String },
}

Three things every variant declares

  1. resolve_target(snapshot, owner) — does it map to a TreeNodeId right now?
  2. descriptor_kind() — which lane in the dependency graph?
  3. requires_rebind_on_structural_change() — must any structural edit force re-binding?
Theoretical lens. Direct/Projection are identity references: deterministically resolvable from the snapshot alone. Relative/Sibling are contextual: resolution depends on the owner’s position in the tree. The four carriers (HostSensitive, CapabilitySensitive, ShapeTopology, DynamicPotential) are residuals: they cannot resolve in the local sequential evaluator, and they emit runtime effects.

Resolution & rebind policy at a glance

ReferenceKindRebind
DirectNodeStaticDirectno
ProjectionPathStaticDirectno*
RelativePathRelativeBoundyes
SiblingOffsetRelativeBoundyes
HostSensitiveHostSensitiveyes
CapabilitySensitiveCapabilitySensitiveyes

* ProjectionPath is path-keyed, but the path index lives inside the snapshot, so resolution is by-id once cached. The W026 rebind floor lists path as static-direct.

Resolving a RelativePath

// formula.rs
match base {
    SelfNode       => Some(owner_node_id),
    ParentNode     => snapshot.parent_id_of(owner_node_id),
    Ancestor(k)   => snapshot.nth_ancestor_of(owner_node_id, k),
}
.and_then(|base| snapshot.try_resolve_descendant_path(base, &segments))
Why the rebind flag matters. Suppose a leaf moves in the tree. A DirectNode reference still names the same id and still resolves; a RelativePath reference may now point somewhere different. The flag is consumed by the dependency-descriptor diff that powers DependencyAdded/DependencyRemoved/DependencyReclassified seeding.

From TreeFormula to DependencyDescriptor

Every node that calculates carries a TreeFormulaBinding: an owner_node_id, immutable artifact handles, and an expression tree. The expression is small and deliberate, so OxCalc can reason about it without re-implementing OxFml.

pub enum TreeFormula {
    Literal      { value: String },
    Reference    (TreeReference),
    Binary       { op, left, right },
    FunctionCall { function_name, arguments,
                    may_introduce_dynamic_dependencies },
    RawOxfml     { source_text, reference_carriers },
}

Lowering rules

  • One descriptor per Reference site, with a deterministic descriptor_id like bind:formula:leaf:ref:0.
  • Literal emits nothing. Binary recurses into both subtrees. FunctionCall recurses into args.
  • RawOxfml uses the explicit reference_carriers list — OxCalc does not re-parse OxFml syntax.
  • The descriptor’s requires_rebind_on_structural_change is copied from the reference shape.

Worked example

A formula that adds the parent’s “Neighbor” to a 2-up-ancestor’s “Sibling” lowers to two relative-bound descriptors:

// expression
Binary(Add,
  RelativePath { base: ParentNode,    seg: ["Neighbor"] },
  RelativePath { base: Ancestor(2), seg: ["Sibling"] }
)

// lowered descriptors (in path-resolution order)
DependencyDescriptor {
    descriptor_id:        "bind:formula:leaf:ref:0",
    owner_node_id:        node:4,
    target_node_id:       Some(node:5),  // Neighbor
    kind:                 RelativeBound,
    carrier_detail:       "relative_path:ParentNode:Neighbor",
    requires_rebind_on_structural_change: true,
}
DependencyDescriptor {
    descriptor_id:        "bind:formula:leaf:ref:1",
    owner_node_id:        node:4,
    target_node_id:       Some(node:3),  // Sibling
    kind:                 RelativeBound,
    requires_rebind_on_structural_change: true,
}

Diagnostics emerge here

If a reference cannot resolve at lowering time, the descriptor still exists — but with target_node_id = None. This becomes a graph diagnostic downstream (UnresolvedReference, HostSensitiveReference, …) rather than silently dropping the dependency.

Determinism

Sorting by descriptor_id after lowering is deliberate: every downstream pass — graph build, invalidation closure, evaluation order — is sensitive to ordering. Determinism here is a load-bearing invariant for diff-stable trace artifacts in TraceCalc.

§07 · Graph internals

The DependencyGraph in detail

One graph per snapshot. It is built once per recalc run and never mutated thereafter.

pub struct DependencyGraph {
    pub snapshot_id:           StructuralSnapshotId,
    pub descriptors_by_owner:  BTreeMap<_,Vec<_>>,
    pub edges_by_owner:        BTreeMap<_,Vec<DependencyEdge>>,
    pub reverse_edges:         BTreeMap<_,Vec<_>>,
    pub cycle_groups:          Vec<Vec<TreeNodeId>>,
    pub diagnostics:           Vec<DependencyDiagnostic>,
}

Cycle detection: Tarjan’s SCC

Cycle groups are produced by Tarjan’s strongly-connected-components algorithm directly over edges_by_owner. Singletons that don’t self-loop are filtered out — only true cycles survive.

fn strong_connect(node, edges, &mut state) {
    state.indices.insert(node, state.index);
    state.lowlinks.insert(node, state.index);
    state.index += 1;
    state.stack.push(node);
    state.on_stack.insert(node);

    for edge in edges_for(node) {
        if !state.indices.contains_key(&edge.target) {
            strong_connect(edge.target, …);
            state.lowlinks[node] = min(state.lowlinks[node],
                                       state.lowlinks[edge.target]);
        } else if state.on_stack.contains(&edge.target) {
            state.lowlinks[node] = min(state.lowlinks[node],
                                       state.indices[edge.target]);
        }
    }

    if indices[node] == lowlinks[node] {
        // pop SCC
    }
}

Diagnostic taxonomy

Every descriptor whose target cannot resolve becomes a typed diagnostic, not a silent drop:

  • MissingOwner, MissingTarget — referential integrity
  • UnresolvedReference, HostSensitiveReference
  • DynamicPotentialReference, CapabilitySensitiveReference, ShapeTopologyReference

Three indexes, one truth

Forward edges drive evaluation order; reverse edges drive invalidation closure.

Why both directions?

  • Forward (edges_by_owner) is what topological evaluation needs: who must compute before me?
  • Reverse (reverse_edges) is what invalidation closure needs: when X publishes, who is dependent?

Cycle handling

Stage 1 cannot evaluate a cycle. The local engine raises LocalTreeCalcError::CycleDetected and the coordinator records a SyntheticCycleReject. Cycle members in the closure are stamped as NodeCalcState::CycleBlocked so dependents know why they cannot progress.

Snapshot-keyed identity

The graph carries snapshot_id by value. There is no scenario where a graph from one snapshot is reused for a different snapshot — every recalc rebuilds from descriptors derived against the exact snapshot the run targets.

§08 · Invalidation closure

Seeds → closure → calc-state

An invalidation closure is the deterministic answer to “given these dirty seeds, exactly which nodes are impacted, in what order, with what calc state, and which require a rebind?”

Seed reasons

ReasonInitial stateRebind?
UpstreamPublicationNeededno
StructuralRecalcOnlyDirtyPendingno
StructuralRebindRequiredDirtyPendingyes
DependencyAddedDirtyPendingyes
DependencyRemovedDirtyPendingyes
DependencyReclassifiedDirtyPendingyes

Cycle members override: any seed inside a cycle group is stamped CycleBlocked.

Closure algorithm

// dependency.rs (paraphrased)
fn derive_invalidation_closure(seeds) -> InvalidationClosure {
    for seed in seeds {
        record(seed.node, derive_seed_state(seed.reason, in_cycle));
        queue.push(seed.node);
    }

    while let Some(target) = queue.pop_front() {
        for edge in reverse_edges[target] {
            mark(edge.owner, UpstreamPublication);
            if in_cycle(edge.owner) { state = CycleBlocked; }
            if queued.insert(edge.owner) {
                impacted_order.push(edge.owner);
                queue.push_back(edge.owner);
            }
        }
    }
}

BFS over reverse_edges; every visited node is folded into a single NodeInvalidationRecord with merged reasons.

Worked example

Cycle interaction

// dependency.rs test (paraphrased)
let closure = graph.derive_invalidation_closure(&[InvalidationSeed{
    node_id: TreeNodeId(2),
    reason: StructuralRebindRequired,
}]);
// 2 ↔ 3 form a cycle group; 4 → 2 is acyclic
assert_eq!(closure.records[&node:2].calc_state, CycleBlocked);
assert!(closure.records[&node:2].requires_rebind);
assert_eq!(closure.records[&node:4].calc_state, DirtyPending);
Reasons accumulate. A node can be marked for several reasons (e.g. both DependencyRemoved and DependencyAdded when a binding swap re-routes a dependency). All are kept; downstream policy decides what to do.

Where it plugs in

The closure is consumed by treecalc.rs’s rebind gate: if any node in the evaluation order has requires_rebind=true, the entire run is rejected with StructuralRebindRequired rather than producing a partial publication.

§09 · Recalc state machine

Eight node states. One transition table.

Every formula-bearing node walks a strict state machine on every recalc. The Stage1RecalcTracker enforces the legal transitions and maintains the matching overlays.

Solid blue = forward evaluation; green = published; red = reject path; brown = repair re-entry; dashed orange = cycle override.

Per-state overlay protection

Every transition writes an InvalidationExecutionState overlay marked is_protected = true. Overlays only become eviction-eligible after publish_and_clear or release_and_evict_eligible.

fn begin_evaluate(node, basis) {
    require_state(node, Needed)?;
    set_state(node, Evaluating);
    protect_execution_overlay(node, "evaluating");
    upsert_overlay(CapabilityFenceAttachment(basis), …);
}

Required preconditions

  • begin_evaluate requires Needed
  • verify_clean requires Evaluating
  • produce_candidate_result requires Evaluating
  • publish_and_clear requires PublishReady
  • release_and_evict_eligible requires Clean | VerifiedClean

Wrong-state attempts return RecalcError::WrongState{ expected, observed }. No silent advance.

§10 · Overlay lifecycle

Six overlay kinds. Two protection bits. One eviction discipline.

Overlays are how OxCalc carries runtime facts that aren’t structural truth — invalidation state, dynamic dependency edges, capability fences, runtime-effect projections — without smuggling them into the snapshot.

pub enum OverlayKind {
    InvalidationExecutionState,   // per-state lifecycle marker
    DynamicDependency,            // late-bound edges
    ExecutionRestriction,         // host/capability blockers
    ShapeTopology,                // shape-sensitive carrier
    CapabilityFenceAttachment,    // fence the run executed under
    ObserverPriorityMetadata,     // pinned-reader hints
}

pub struct OverlayKey {
    pub owner_node_id:           TreeNodeId,
    pub overlay_kind:            OverlayKind,
    pub structural_snapshot_id: StructuralSnapshotId,
    pub compatibility_basis:    String,
    pub payload_identity:       Option<String>,
}

pub struct OverlayEntry {
    pub key:                  OverlayKey,
    pub is_protected:         bool,
    pub is_eviction_eligible: bool,
    pub detail:               String,
}

Why two protection bits?

  • is_protected — overlay is load-bearing for the current run; do not evict.
  • is_eviction_eligible — overlay may be evicted on the next sweep.

A protected overlay can be promoted to eligible by release_and_evict_eligible; this is how a clean node releases its execution-state overlay back to the eviction sweeper.

Lifecycle: a single node

Eviction predicate

fn evict_eligible_overlays(&mut self) -> usize {
    let to_evict = self.overlays.iter()
        .filter(|(_, e)|
            e.is_eviction_eligible && !e.is_protected)
        .map(|(k,_)| k.clone())
        .collect::<Vec<_>>();
    for key in &to_evict { self.overlays.remove(key); }
    to_evict.len()
}
Pinned-reader interaction. The TraceCalc machine defers eviction while any pinned reader remains. When the last pinned reader is unpinned, every clean node’s overlays are first opened (release+eligible), then a sweep runs. This is how we avoid evicting state out from under a stable view.

Compatibility basis

The compatibility_basis field on OverlayKey is a free-form fence string — typically the snapshot id ("snapshot:1") or a profile id. It is what makes “same key, different fence” possible.

The publication path

Atomic publication

pub fn accept_and_publish(&mut self, publication_id) {
    let accepted = self.accepted_candidate.clone()
        .ok_or(MissingAcceptedCandidate)?;

    let mut values = self.published_view.values.clone();
    values.extend(accepted.value_updates.clone());

    let bundle = PublicationBundle {
        publication_id: ...,
        candidate_result_id: accepted.candidate_result_id,
        structural_snapshot_id: self.snapshot.snapshot_id(),
        published_view_delta: accepted.value_updates,
        published_runtime_effects: accepted.runtime_effects,
        trace_markers: vec!["publication_committed"],
    };

    self.published_view = PublishedView {
        snapshot: self.snapshot.clone(),
        publication: Some(bundle.clone()),
        values,
    };
    self.in_flight_candidate = None;
    self.accepted_candidate  = None;
    self.counters = self.counters.increment_publications();
}
Once accept_and_publish returns, the new state is observable in one step. There is no torn intermediate where some node values are old and others are new.

§12 · Stable observation

Published view, pinned readers, deferred eviction

The published view is the only stable answer to “what does node X currently evaluate to?” Pinned readers freeze a copy of it for stable observation.

pub struct PublishedView {
    pub snapshot:    StructuralSnapshot,
    pub publication: Option<PublicationBundle>,
    pub values:      BTreeMap<TreeNodeId, String>,
}

pub struct PinnedPublicationView {
    pub reader_id:       String,
    pub snapshot_id:     StructuralSnapshotId,
    pub publication_id:  Option<String>,
    pub structural_view: PinnedStructuralView,
    pub values:          BTreeMap<TreeNodeId, String>,
}

Three-step pin protocol

  1. pin_reader(view_id) snapshots the live view by-value, returns the PinnedPublicationView.
  2. Reader observes; even after subsequent publications, the pinned values do not change.
  3. unpin_reader(view_id) drops the pin; eviction-eligible overlays may now flow through the next sweep.

Why coordinator-owned, not reader-owned?

A reader-owned snapshot would race the coordinator’s lifetime; a coordinator-owned pin lets the engine defer eviction until it is genuinely safe. The TraceCalc machine implements exactly this contract, and counts the deferred-eviction events as observable evidence.

What an unpin actually triggers

Counters that prove it

The TraceCalc reference machine increments observable counters at every coordinator boundary:

  • reader.pinned, reader.unpinned
  • candidate.admitted, candidate.published, candidate.rejected
  • overlay.release_deferred, overlay.eviction_eligible, overlay_evictions
  • retention_waiting_on_readers

These counters are part of the conformance contract — replays compare them across engines.

Stable observer rule. A pinned view never sees a “torn” combination of new structural state with old values, or vice versa: both sides of a publication transition together.

§13 · End-to-end run

LocalTreeCalcEngine::execute — the whole pipeline

This is the inner loop that ties everything together. One method; thirteen explicitly timed phases.

Each phase records its duration; the artifacts bundle is the canonical handoff to TraceCalc and consumer hosts.

Three result states

  • Published — at least one node’s value changed; coordinator emitted a bundle.
  • VerifiedClean — all candidate values matched the seeded published values; no publication.

§14 · The OxFml seam

How OxCalc consumes OxFml

OxCalc owns recalc, coordination, and publication. OxFml owns formula syntax, binding, and runtime evaluation. They meet at exactly one bridge: the MinimalUpstreamHostPacket.

OxCalc translates its TreeFormula into A1-flavoured OxFml source, hands the packet to OxFml, and validates the commit bundle on the way back.

The packet, in shape

pub struct MinimalUpstreamHostPacket {
    pub formula_slot:        MinimalFormulaSlotFacts,
    pub binding_world:       MinimalBindingWorld,
    pub typed_query_facts:   MinimalTypedQueryFacts,
    pub runtime_catalog:     MinimalRuntimeCatalogFacts,
    pub publication_context: Option<_>,
}
  • formula_slot — formula text, stable id, channel kind, caller anchor.
  • binding_world — synthetic cell fixture & defined-name bindings (one per resolved TreeReference).
  • typed_query_facts — host info / RTD modes; how to refuse residuals deterministically.
  • runtime_catalog — optional library-context snapshot.

Translation discipline

Each TreeReference becomes a synthetic A1 token:

// treecalc.rs · TranslationState::bind_target
let token = format!("TREE_REF_{}_{}",
                    self.owner_node_id.0,
                    self.next_reference_index);

And each working value becomes a synthetic A1 cell at A<node_id>. Residual references emit OxFml call-throughs that deliberately fail in deterministic ways:

HostSensitive       => "INFO(\"system\")"
CapabilitySensitive => "INFO(\"osversion\")"
ShapeTopology       => "ROWS(A1:A1)"
DynamicPotential    => "RTD(\"TREECALC\",\"\",topic)"

OxFml then meets them with a typed query bundle that surfaces a HostInfoError::ProviderFailure or RtdProviderResult::CapabilityDenied — the seam handles errors, not silent fallbacks.

Bundle validation

The post-run bundle must agree with the candidate. Three identity checks must pass:

  1. candidate_result_id match
  2. formula_stable_id match
  3. commit_value_delta.candidate_result_id match

A mismatch surfaces as OxfmlCommitBundleIncompatible and the run rejects.

What the local engine cannot evaluate, it surfaces

OxCalc’s local sequential engine is intentionally narrow. It evaluates structurally-resolvable references and plain arithmetic / built-in functions. Stage 2/3 lanes will accept many of these natively; today they are first-class evidence that a node cannot publish in this run.

From residual to overlay

fn build_runtime_effect_overlays(input, owner, effects) {
    effects.iter().enumerate().map(|(i, e)|
        OverlayEntry {
            key: OverlayKey {
                owner_node_id: owner,
                overlay_kind:  runtime_effect_overlay_kind(e),
                structural_snapshot_id: input.snapshot.id(),
                compatibility_basis:    input.basis.clone(),
                payload_identity:       Some(format!(
                  "{}:runtime_effect:{i}",
                  input.candidate_result_id)),
            },
            is_protected:         true,
            is_eviction_eligible: false,
            detail:               format!("{}|{}", e.kind, e.detail),
        })
    .collect()
}

§16 · OxFml capability surface uptake

How OxCalc tracks which OxFml capabilities it consumes

The seam from page 15 says how OxCalc talks to OxFml. This page is about the parallel question: which OxFml capabilities are actually behind direct OxCalc evidence, and how that set is grown over time. The vocabulary that answers it — surface, uptake, ledger, promotion decision, exact blocker — was the load-bearing workflow for worksets W039 through W045.

The three terms of art

Capability surface. The set of OxFml-exposed capabilities (formula evaluation, callable metadata, registered-external providers, publication/display payloads, runtime-facade types, conditional-formatting typed rules, …) for which OxCalc has wired a consumer and exercised it under fixtures. Anything OxFml offers that OxCalc has not exercised is off-surface, even if it is implemented upstream.

Uptake. The act of promoting a capability from "observed" to "consumed under direct evidence." Uptake is gated by a strict evidence shape — not by the existence of the upstream feature.

Ledger. A schema-stamped JSON evidence packet that records the uptake decision for a given workset. Two ledger families collaborate: a release-grade ledger maps residual lanes to successor obligations, and an OxFml-seam ledger records surface registers and the per-workset promotion decision.

The four disposition kinds

Every row in a consumed-surface register carries one of these:

  • exercised_current_surface — promoted under direct evidence for this run.
  • exercised_current_surface_promoted — same as above, with broader closure claimed.
  • note_level_watch — acknowledged from upstream notes, not yet exercised.
  • exact_remaining_blocker — named, sourced, and carried forward as an obligation.

A row never silently disappears. A blocker is either resolved (and the row flips to exercised_current_surface_promoted) or it carries forward to the next workset's register by name.

The handoff_watch rule. An OxFml handoff packet may be filed only when an exercised OxCalc fixture exposes a concrete mismatch with OxFml. Speculation does not generate a handoff. This is what keeps the seam quiet under normal operation and noisy only on real drift.

The schema-constant rolling pattern

Each workset's runner declares its own schema constants and artifact paths. src/oxcalc-tracecalc/src/oxfml_seam.rs shows the pattern, one constant family per workset:

§17 · W046 redirect & calc-aylq successors

From ledger uptake to handler-law model — and what carried forward

After W045's non-promotion close, the planned W046 was originally another tranche of the same ledger-driven promotion workflow. A showcase review then redirected W046 entirely. The workflow that named blockers row by row gave way to a workflow that models the seam as engine effects with handler-law constraints. The remaining "broader uptake" work did not vanish — it was reclassified as carry-forward residual and routed to a successor epic.

The W046 redirect

The active workset is W046_CORE_FORMALIZATION_ENGINE_SEMANTIC_PROOF_SPINE.md (the earlier …_PROOF_SERVICE_CONSOLIDATION_AND_RELEASE_GRADE_PROMOTION_READINESS draft is archived). The governing shift is stated bluntly:

  1. Start with engine semantics, not promotion classification.
  2. Convert the showcase engine catalog into OxCalc-owned spec objects.
  3. Build formal targets for execution phases before claiming service or release readiness.
  4. Use TraceCalc as the correctness oracle; replay TreeCalc/CoreEngine against it.
  5. Keep closure artifacts as coverage accounting, not a substitute for semantic models.

The OxFml seam under the redirect

Bead calc-gucd.7 produced W046_OXFML_SEAM_LET_LAMBDA_FORMATTING_PUBLICATION_AND_CALLABLE_BOUNDARY_MODEL.md: an effect-signature spec with handler-law obligations rather than a per-row uptake register.

§18 · TraceCalc

The reference machine that watches the engine

TraceCalc is OxCalc’s second implementation of the same coordinator-and-overlay semantics, driven by JSON scenarios. It is how we prove the engine and reference agree, step by step, counter by counter, event by event.

Anatomy of a scenario

// from contracts.rs

    pub schema_version:   String,
    pub scenario_id:      String,
    pub calc_space:       String,
    pub initial_graph:    TraceCalcInitialGraph,
    pub initial_runtime:  TraceCalcInitialRuntime,
    pub steps:            Vec<TraceCalcStep>,
    pub expected:         TraceCalcExpected,
    pub replay_projection: Option<_>,
    pub witness_anchors:  Option<_>,
    ...
}

Step kinds

  • pin_view / unpin_view
  • mark_stale — seed dirty nodes
  • admit_work — start a candidate
  • emit_candidate_result — values + shape updates + effects
  • emit_reject — typed reject path
  • publish_candidate — accept_and_publish
  • verify_clean — value-equal seed path
  • seed_overlay — pre-load an overlay
  • reset_fixture — wipe between phases

Two machines, one semantics

The crate exposes TraceCalcEngineMachine (engine-driven) and TraceCalcReferenceMachine (the gold-standard reference). Both call the same execute_shared; conformance lanes diff their artifacts.

What gets compared

Witness anchors

Witness anchors give scenarios stable handles for downstream proof work — phase blocks, event groups, reject records, view slices. They are how a Lean obligation can refer to “the third reject in this scenario’s phase block 2” without copying contents.

§19 · Formal assurance

TLA+ and Lean: the engine’s second pair of eyes

OxCalc maintains formal models alongside the implementation. They are not decorative — they fail when the engine drifts, and they unblock when promotion ledgers say so.

TLA+ — coordinator state model

The formal/tla/CoreEngineStage1.cfg model formalizes the Stage 1 coordinator state machine: snapshot id, in-flight candidate, accepted candidate, published view, pinned readers, reject log. Safety properties include:

  • no publication without a recorded accepted candidate
  • candidate snapshot id matches coordinator snapshot id
  • reject log monotonically grows; published view never silently shrinks
  • pinned reader values do not change after the pin

Lean — typed kernel obligations

formal/lean/OxCalc/CoreEngine/Stage1State.lean begins the typed obligation tree for Stage 1. The intention is to prove, against a Lean-faithful state model, the same invariants TLA+ checks operationally. Bridges to the implementation come through the conformance scaffolding.

Replay-grade artifacts

Every TraceCalc scenario produces a deterministic artifact set. Replays land under formal/replay/ and docs/test-runs/. A small example:

Five things consumer hosts get for free

  1. Deterministic artifacts for every request — the dependency graph, invalidation closure, evaluation order, runtime effects, and overlays are part of the response.
  2. Pinned-reader stability — the host can pin a view id, observe, and trust that pinned values do not move under it.
  3. Typed reject paths — an eight-class taxonomy that maps host failures, capability denials, cycles, and fence drifts to distinct codes.
  4. OxFml seam isolation — the host never sees OxFml directly; it sees only the translated formulas, validated commit bundles, and runtime effects.
  5. Phase timings — every recalc returns a microsecond breakdown of the thirteen phases so hosts can profile in production-grade telemetry.
What changes between stages. Hosts see more concurrency in the runtime effects log, fewer rejects on dynamic-dependency carriers, and lower wall-time on partitionable workloads — but the same artifact shapes and the same publication semantics.

Compatibility basis is your contract

Every overlay, candidate, and publication carries a compatibility_basis. Treat it as a versioned fence string: changing it forces the engine to rebuild from a clean slate. It is the cheapest way for a host to invalidate everything cleanly.

Five takeaways

  1. Identity is sacred. Everything else — projection paths, overlays, runtime effects — is derived from snapshot-keyed identity.
  2. One publisher. Coordinator authority is a correctness property, not a coordination convenience. Evaluators never publish.
  3. OxFml is a seam, not a leak. OxCalc translates, hands over a packet, validates the bundle, and goes back to its own world. There is exactly one bridge.
  4. Residuals are first-class. What the local engine can’t evaluate becomes typed evidence — never silent failure.
  5. The engine and its reference machine watch each other. TraceCalc’s eight mismatch kinds catch drift before it becomes incident.
