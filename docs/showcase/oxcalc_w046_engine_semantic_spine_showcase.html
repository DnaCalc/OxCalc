1. The W046 verdict

What improved

We now have a named semantic model for the engine’s execution phases, with Lean/TLA artifacts and replay/checker roots attached to concrete engine concepts rather than only readiness labels.

What it means

The calculation engine can now be explained as a small-step transition system with graph/dataflow invariants, phase authority boundaries, observable refinement targets, and proof-carrying evidence.

What it does not mean

W046 is not full verification. It is a coherent proof spine with selected checked artifacts, bounded models, selected trace validation, exact blockers, and successor routing.

The main intellectual gain: W046 makes engine correctness decomposable. Instead of asking “does the calculator work?”, we can ask: did graph construction preserve edge facts, did invalidation cover all reverse-reachable dependents, did order select dependencies before dependents, did reads use only stable/prior values, did rejection prevent publication, did OxFml effects stay outside publication authority, and did emitted artifacts carry enough facts to check these claims?

2. The heart of the calc engine

At its core, OxCalc is an incremental dataflow engine around a structural tree. Formulas are attached to nodes; formulas lower into dependency descriptors; descriptors become a dependency graph; edits seed invalidation; invalidation selects affected formulas; formulas are evaluated in dependency order using stable or prior values; results become candidates; the coordinator either publishes atomically or rejects without mutating the published view.

Structural snapshotTree nodes, symbols, parents, formulas, constants, bind artifacts, snapshot id.
Formula catalogTree formula bindings by owner node; OxFml preparation inputs.
DescriptorsStatic, relative, dynamic, host/capability/topology-sensitive or unresolved references.
Dependency graphForward edges, reverse edges, diagnostics, cycle groups/SCCs.
InvalidationSeeds and reverse-reachability closure; rebind flags and dirty records.
Order & readsTopological formula order; stable published inputs plus prior computed updates.
EvaluateOxFml/TreeCalc formula execution produces values, diagnostics, runtime effects, or typed failures.
Candidate / rejectAcceptedCandidateResult or RejectDetail; cycle/dynamic/rebind failures route to no-publish.
Publish / traceCommitBundle atomically updates published view; traces/replay artifacts expose facts.

3. Execution as phase authority

W046’s clearest engine insight is that each phase has authority over only some facts. Correctness depends on preventing authority leaks.

6. The formalization angles: seven lenses on one engine

Graph/dataflow

Dependency edges define dataflow. Reverse edges define invalidation flow. SCCs define where acyclic evaluation stops and cycle rejection begins.

  • edge target soundness
  • reverse-edge converse
  • cycle group support
  • closure reachability
Operational semantics

The engine is a small-step machine: BuildGraph, CloseInvalidation, MarkDirty, SelectOrder, Evaluate, Accept, Reject, Publish, EmitTrace.

  • pre/post conditions
  • terminal exclusivity
  • no publish from rejected state
Type/effect boundary

OxFml evaluation can request effects; OxCalc owns handlers for graph, invalidation, candidate, reject, and publication.

  • OxFml cannot publish directly
  • dynamic effects route through rebind/reject
  • format/display are distinct from value publication
Refinement

TraceCalc is a selected reference machine; TreeCalc/CoreEngine traces are implementation observations to compare against observable values, diagnostics, rejects, publications, and dependency effects.

Proof-carrying traces

Important runs should emit or reference enough facts that a smaller checker can validate semantic properties without trusting the full engine.

Rust bridge

W046 maps implementation fields and emitted artifacts back to semantic facts, then checks selected traces and in-memory invariants.

Scale semantics

Performance profiles become semantic-regression signatures only when closed-form values and graph/invalidation/dynamic counts pass. Timing alone is never correctness.

Consequence matrix

Readiness lanes are downstream consequences. W046 keeps them honest: no promotion without direct semantic or checker-backed evidence.

Closure audit

The final audit maps every W046 promise to concrete specs, formal artifacts, test roots, blockers, and W047 successor beads.

8. Dependency graph: the first semantic hinge

The engine starts becoming formal when formula references stop being strings and become graph facts.

descriptor(owner, maybe_target, kind, carrier_detail)
  ↳ if target exists: edge(owner, target, kind)
  ↳ if target absent/dynamic/host-sensitive: diagnostic(owner, kind, detail)

reverse_edges[target] contains edge(owner,target)
cycle_groups = SCCs or self-loops over formula edges

This is the hinge between syntax and execution. Once descriptors are graph facts, invalidation and evaluation can be stated as graph/dataflow properties rather than incidental implementation behavior.

9. Invalidation: change as reverse reachability

Invalidation is not “rerun everything.” It is a reverse-edge closure from seeds.

Seed classMeaningExpected consequence
structural recalcnode value changed or structure affects formuladirty impacted formulas
upstream publicationdependency published new valuepropagate to dependents
dependency added/removed/reclassifiedgraph shape changedrebind/reject if stale
dynamic potentiallate-bound reference unresolveddiagnostic or reject/no-publish

The important formal target is no-under-invalidation: if a dependent is reverse-reachable from an invalidation seed, it must not be silently skipped.

10. Evaluation order and read discipline

Evaluation is safe only if each formula reads a stable published input or a prior computed formula value.

for edge(owner, target):
  if owner and target are both formula nodes:
    target appears before owner in evaluation_order

read(owner, target) is legal iff:
  target is stable published input
  OR target was already computed earlier in evaluation_order

This is the core execution law. It is why topological order is not an implementation detail; it is the semantic condition that prevents future reads and torn candidates.

11. Candidate, reject, publication

The coordinator is the publication authority. Formula evaluation proposes; coordinator disposes.

TerminalFacts requiredPublished view
publishedaccepted candidate, target set, value delta, compatible basisatomically updated
verified cleanno changed values; order/closure facts still coherentunchanged
rejectedtyped reject detail, no publication bundleunchanged

W046 repeatedly checks the no-publish law: cycles, dynamic failures, stale rebind, diagnostics, and host-sensitive effects cannot mutate the published view directly.

12. Proof-carrying traces: the engine explains itself

The checker added in W046 is small but strategically important. It turns emitted artifacts into independently checkable semantic evidence.

TreeCalc facts checked
  • graph artifact parseable
  • edge/descriptor links
  • derived reverse index
  • order covered by closure/reject
  • formula edges respect order
  • stable/prior observations
  • candidate/publication consistency
  • reject has no publication
  • dynamic potential rejects
TraceCalc facts checked
  • sidecars parse
  • normalized event families exist
  • required equality surfaces have artifacts
  • cycle region rejects
  • cycle trace has no publication
  • assertions/conformance clean
Why this matters

Proof-carrying traces decouple expensive execution from cheaper validation. Future optimized engines can emit semantic facts, and a checker can reject runs whose facts violate the kernel.
