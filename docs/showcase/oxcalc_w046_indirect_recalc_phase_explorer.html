Important honesty boundary: the exact spreadsheet text INDIRECT("A" & B1) is not natively parsed as grid syntax today. This document therefore separates reference semantics from current TreeCalc representation: TraceCalc can express the dynamic dependency switch as an oracle-style scenario; TreeCalc can represent resolved/unresolved dynamic references and dependency shape updates, but not yet the string-to-name resolution pipeline from that formula text.

Simple calculation

A1 = 2
B1 = 3
C1 = A1 + B1

edit: A1 := 5  (interactive default)
result: C1 = 8

Only a value changes. The dependency graph remains C1→A1 and C1→B1. Invalidation starts at A1 and reaches C1. Evaluation order is just C1. Publication changes C1.

Complex calculation

A1 = 3
A2 = 4
B1 = 1
C1 = A1 + INDIRECT("A" & B1)
D1 = INDIRECT("C1")

edit: B1 := 2
before: C1 = 3 + A1 = 6, D1 = 6
after:  C1 = 3 + A2 = 7, D1 = 7

A value edit changes a dynamic address. That changes graph shape: C1 releases dynamic edge to A1 and activates dynamic edge to A2. Then C1's value changes, so D1 must recompute.

Through the W046 lenses

1. Dependency graph

Simple: graph shape stable. Complex: graph shape changes because C1's dynamic descriptor changes from A1 to A2. D1 keeps a dynamic edge to C1.

2. Reverse invalidation

Simple: A1 reverse edge reaches C1. Complex: B1 must reach C1 through the dynamic-address expression dependency; then C1 reaches D1 through value dependency.

3. Rebind / soft refs

Complex edit is not merely dirty-value recalc. It requires re-evaluating a name expression and updating dynamic dependency shape before publication.

4. Evaluation order

Simple order is C1. Complex post-edit order is C1 then D1. C1 must compute before D1 because D1 reads C1.

5. OxFml effects

INDIRECT is an effectful reference-resolution request. OxFml should not publish; it asks OxCalc to resolve/read/emit dependency effects.

6. Reject boundary

If the dynamic name cannot be resolved, current TreeCalc should reject/no-publish. That is safer than publishing a value with an unknown graph shape.

7. Trace evidence

A proof-carrying trace should record old dynamic edge release, new dynamic edge activation, invalidation closure, order C1→D1, candidate values, and publication.

8. Refinement

TreeCalc can refine TraceCalc if observable values, dependency shape updates, runtime effects, reject/publish decision, and trace labels match.

9. Optimization

An optimized engine can skip re-evaluating unaffected formulas, but cannot skip dynamic target resolution when selector B1 changes.

Future optimization/remodeling ideas exposed by this case

Incremental dynamic dependency engine

Represent INDIRECT("A" & B1) as two layers: address expression depends on B1; resolved edge points to A1/A2. On B1 edit, recompute the address expression first, then patch graph edges.

Two-tier invalidation

Separate value invalidation from shape invalidation. B1 edit causes C1 shape invalidation; C1 publication causes D1 value invalidation.

Effect handlers

Make ResolveDynamic(owner, text, context) explicit. Handler returns either target node + dependency update, or typed reject/no-publish.

Proof-carrying dynamic sidecars

Emit old target, new target, selector input, release/activate events, reverse-edge patches, and order proof. Then checker can validate dynamic rebinding without trusting the evaluator.

Optimization danger

Caching C1 just because A1 did not change is wrong. B1 changed the address expression, so C1's dependency and value changed.

Safe optimization target

D1's dynamic target remains C1, so D1 does not need dynamic rebind; it only needs value recomputation after C1 publishes. That distinction is optimization gold.
