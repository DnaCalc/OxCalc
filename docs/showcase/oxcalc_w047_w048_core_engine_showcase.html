System map

W047 and W048 are one pipeline, not two isolated features

W047 established the effective-graph overlay model. A formula can discover at calculation time that it depends on a new node, no longer depends on an old node, or changed the class of a runtime-derived dependency. Those facts do not mutate the structural snapshot. They form O_candidate.

W048 then asks what happens when a structural graph or a candidate overlay graph contains a cycle. The answer is now explicit: materialize the relevant graph layer, classify SCCs, reject non-iterative cycles with no publication, and preserve evidence for replay and future formalization.

Core shape now: structural graph build → published overlay composition → invalidation → evaluation → candidate overlay diff → candidate graph cycle classification → atomic publish or no-publish reject.
The W047 overlay pipeline creates the W048 candidate-effective graph classification point.
LayerQuestion answeredCurrent status
G_structWhat follows from the immutable structural snapshot?Implemented in TreeCalc graph build with forward/reverse edges and SCC groups.
G_effWhat graph is effective after previously accepted runtime overlay facts?Represented by seeded published runtime effects and materialized W048 graph sidecars.
G_eff_candidateWhat would be effective if this candidate overlay published?Classified for cycle/reject/no-overlay-commit behavior in W048 artifacts.

W047 state model

CTRO: runtime graph shape without structural mutation

State vocabulary

SymbolMeaning
SImmutable structural snapshot: nodes, formulas, names, and static binding facts.
G_structDependency graph derived only from S.
O_publishedAccepted runtime overlay from prior publications.
G_effComposed graph: compose(S, G_struct, O_published).
O_candidateCandidate-local runtime overlay facts from the current attempt.
G_eff_nextCandidate effective graph for cycle classification and repair.

Delta taxonomy

  • activate_dependency: new runtime dependency observed.
  • release_dependency: prior runtime dependency not observed now.
  • reclassify_dependency: same edge, different dependency class.
  • activate_region / release_region: future grid/spill-shaped dependency/output changes.
  • cycle_introduction: candidate graph creates or exposes an SCC.
  • unsupported_or_unresolved: insufficient facts; reject/fallback rather than publish unsafely.
Design constraint. CTRO never promotes runtime facts into structural truth. Runtime graph truth only advances through an accepted publication bundle.
// Conceptual transition
G_eff       = compose(S, G_struct, O_published)
frontier    = invalidate(G_eff, seeds)
EvalResult  = evaluate(frontier, values)
O_candidate = diff(runtime_effects, O_published)
G_eff_next  = compose(S, G_struct, O_candidate)
if cycle_or_unsafe(G_eff_next): reject(no_overlay_commit)
else publish(value_delta + overlay_delta)

pub enum InvalidationReasonKind {
  StructuralRebindRequired,
  StructuralRecalcOnly,
  UpstreamPublication,
  DependencyAdded, DependencyRemoved, DependencyReclassified,
  DynamicDependencyActivated,
  DynamicDependencyReleased,
  DynamicDependencyReclassified,
}

Why this matters

W047 added CTRO-specific invalidation reasons that are dirtying reasons but not structural rebind gates. A dynamic dependency activation or release means the owner and downstream dependents need recalculation; it does not imply the formula binding itself must be rebuilt.

Key distinction: DependencyAdded forces rebind; DynamicDependencyActivated marks work repairable through the CTRO path.
fn reason_requires_rebind(reason: InvalidationReasonKind) -> bool {
  matches!(reason,
    StructuralRebindRequired | DependencyAdded |
    DependencyRemoved | DependencyReclassified)
}

fn dynamic_dependency_shape_updates(published, graph) -> Vec<DependencyShapeUpdate> {
  current = dynamic_dependency_facts_from_graph(graph)
  for released in published - current: emit("release_dynamic_dep")
  for activated in current - published: emit("activate_dynamic_dep")
  sort_by(kind, affected_node_ids)
}

W048 graph model

Circular references are classified over materialized graph layers

W048 treats cycles as graph facts that must be visible at the layer where they are discovered. The same SCC classification vocabulary is used for structural cycles, published-overlay cycles, and candidate-overlay cycles.

structural

G_struct

Formula text and static binding facts already contain an SCC. Non-iterative Stage 1 rejects before ordinary acyclic evaluation.

published effective

G_eff

A previously accepted overlay edge participates in the current effective graph. Invalidation and cycle diagnostics see it as graph truth for this run.

candidate effective

G_eff_candidate

A current candidate would create an SCC if published. The candidate is rejected and the overlay is not committed.

cycle_sourceWhere the SCC came from: structural, published overlay, or candidate overlay.
member_orderDeterministic sorted member order for replay and diagnostics.
internal_edgesEdges that keep the SCC closed.
incoming_boundary_edgesDependents or precedents at the cycle-region boundary.

Cycle data structures

Forward/reverse edges plus Tarjan SCCs are the current core

TreeCalc builds forward dependency edges from lowered descriptors, derives reverse edges for invalidation, then runs a deterministic Tarjan pass over all snapshot nodes. Self-loops are cycle groups, not just multi-node SCCs.

  • Forward edge: owner formula → precedent target.
  • Reverse edge: target → dependent owner.
  • Cycle group: non-trivial SCC or self-loop.
  • Invalidation uses reverse edges and changes node state to CycleBlocked when needed.
fn find_cycle_groups(snapshot, edges_by_owner) -> Vec<Vec<TreeNodeId>> {
  // Tarjan SCC over deterministic node order
  for node_id in snapshot.nodes().keys() {
    if !indices.contains(node_id) { strong_connect(node_id) }
  }
  if group.len() > 1 || is_self_cycle(node_id) {
    group.sort(); groups.push(group)
  }
  groups.sort(); groups
}
fn derive_invalidation_closure(seeds) -> InvalidationClosure {
  cycle_members = flatten(graph.cycle_groups)
  queue = seeds
  while target = queue.pop_front() {
    for reverse_edge in graph.reverse_edges[target] {
      mark(dependent, UpstreamPublication)
      if cycle_members.contains(dependent) { state = CycleBlocked }
      queue.push(dependent)
    }
  }
}
W048 checker obligations: forward/reverse converse, non-empty cycle regions, cycle reject/no publication, and release/re-entry publication for owner plus downstream dependent.

Formalization and checkers

What is formalized now, and what remains successor work

W047 formal/checker position

W047 records CTRO formal targets rather than claiming Lean/TLA closure. The key W049 targets are effective graph composition, no-under-invalidation, reject implies no candidate overlay publication, and atomic value/overlay publication.

W048 formal/checker position

W048 adds a checked model projection over actual TreeCalc artifacts. The executable checker validates graph-layer obligations and cycle publication behavior against the W048 run root.

W048 checked obligations

  • Every forward edge has reverse converse.
  • Every reverse edge has forward converse.
  • Every cycle region is non-empty.
  • Structural and CTRO cycle fixtures reject with SyntheticCycleReject.
  • Rejected cycle fixtures emit no publication bundle.
  • Release/re-entry publishes owner and downstream dependent.

Current state

Where the core engine stands after W047 and W048

What looks strong now

  • Dynamic dependency activation/release is a first-class CTRO path, not a structural-rebind overclassification.
  • Publication bundles can carry value updates and runtime dependency consequences atomically.
  • Cycle behavior is visible across graph layers, not hidden in scheduler fallback.
  • Structural cycles and CTRO-created cycles share one Stage 1 rejection boundary.
  • TraceCalc, TreeCalc, graph sidecars, and checkers align on the predecessor W048 examples.

What is reopened now

  • Excel-match, bit-exact iterative circular calculation is active W048 scope.
  • Legacy Python W048 tooling has PowerShell replacements and must not be used for future W048 closure.
  • Grid/spill and data-table cycle disposition must be exactly scoped or blocked.
  • Full closure cannot rely on routing core circular semantics to W049.
Bottom line. OxCalc has a concrete, evidence-backed Stage 1 model for dynamic dependency rebinding and circular-reference rejection/re-entry. W048 is now reopened to extend that foundation into the comprehensive Excel-match circular-reference solution, including bit-exact iterative calculation.
