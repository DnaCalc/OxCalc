Mental model

W047 creates graph truth at calculation time; W048 makes it safe

Before W047, dynamic formulas could be treated as rebind pressure or opaque host sensitivity. W047 gives them a more precise lane: the evaluator reports runtime facts, TreeCalc compares those facts with the last accepted runtime overlay, and the candidate result carries dependency-shape updates alongside values.

W048 then applies the same safety rule to all graph layers: if a cycle exists under the active profile, the engine either rejects without publication or uses an explicit opt-in iterative profile whose terminal values are backed by Excel observation packets and replay-visible traces.

Core invariant: no value publication is visible without the corresponding accepted overlay consequences, and no rejected candidate overlay is silently promoted.

Important non-claim

Runtime facts are not structural facts. A published dynamic edge affects future invalidation, but it does not rewrite formula binding or mutate the immutable structural snapshot.

W047's key TreeCalc change is not a new graph type; it is a new publication input/output shape. Previously accepted runtime effects seed the current run. The current dependency graph yields current dynamic facts. A deterministic set difference produces release and activation shape updates.

Optimization boundary. TreeCalc emits compact deltas and uses ordinary coordinator publication. TraceCalc records a more explicit reference trace, but both must agree on published surfaces for declared cases.
fn dynamic_dependency_shape_updates(published, graph) -> Vec<DependencyShapeUpdate> {
  current = dynamic_dependency_facts_from_graph(graph)
  for released in published - current:
    emit("release_dynamic_dep", sorted(owner,target))
  for activated in current - published:
    emit("activate_dynamic_dep", sorted(owner,target))
  sort_by(kind, affected_node_ids)
}

W048 graph layers

Cycle detection is graph-layered, deterministic, and replayable

W048 materializes cycle facts across three graph bases: structural, published-effective, and candidate-effective. The SCC detector uses Tarjan over the graph's forward dependency edges and records both non-trivial SCCs and self-loop SCCs.

Excel-match iterative slice

What “Excel-match” means in W048 now

W048 does not implement arbitrary Excel workbook cycle semantics. It implements a declared, falsifiable fixture slice whose values are grounded in black-box Excel observation packets for one observed Excel host/version.

fn publish_excel_match_iterative_cycle(input, coordinator, tracker, graph, closure, ...) {
  let Some((order, value_updates, trace_summary)) =
    excel_match_iterative_fixture_surface(input)
  else { reject(CycleDetected) };
  diagnostics += ["cycle.excel_match_iterative", "cycle_iteration_trace", trace_summary];
  admit_candidate(order, value_updates);
  accept_and_publish();
}

Formalization

Small executable formal surface, not decorative proof text

The current formalization around W048 is intentionally artifact-grounded. It defines graph layers, forward/reverse edges, cycle regions, reject states, no-overlay-commit, and release/re-entry, then checks those obligations against TreeCalc run artifacts.

Current state

Where the core engine work stands now

Integrated now

  • CTRO dynamic activation/release/reclassification.
  • Atomic value + runtime-effect publication.
  • Downstream invalidation for dynamic target switches.
  • Structural and CTRO-created cycle rejection.
  • Release/re-entry after dynamic cycle removal.
  • Declared Excel-match iterative fixture publication.
  • TraceCalc and TreeCalc fixture agreement.
  • Executable graph/formal checker surface.

Documented limitations

  • No broad cross-version Excel compatibility claim.
  • Excel scope is one observed host/version.
  • TreeCalc fixture path is not a universal Excel workbook parser/evaluator.
  • Multithread Excel behavior remains a profile dimension, not the implemented default fixture profile.
  • Full mechanized proofs route to successor formalization.
Bottom line. W047 and W048 moved the engine from “static graph plus conservative cycle rejection” toward a profile-governed calculation core where runtime graph truth is explicit, candidate overlays are safe, cycles are classified over materialized graph layers, and Excel-observed iterative behavior can be admitted under a declared compatibility basis.
