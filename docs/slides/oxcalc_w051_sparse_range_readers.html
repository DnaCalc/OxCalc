OxCalc W051

Sparse Range Readers And Defined-Entry Semantics

The next workset after W050: Excel-compatible whole-column and large-area range reads without dense materialisation.

activation_review_ready

What it is

A concrete sparse reader for worksheet references: declared extent, defined entries, blank reads, point reads, and deterministic iteration.

What it is not

Not generic virtualized arrays, not arbitrary queryable objects, and not a broad rich-data protocol.

Why now

W050 moved OxCalc onto OxFml runtime packages and formal inputs. W051 uses that seam to make large Excel ranges practical.

Source: docs/worksets/W051_SPARSE_RANGE_READERS_AND_DEFINED_ENTRY_SEMANTICS.md1 / 14

Problem

Dense range materialisation is the wrong shape for ordinary large worksheet references.

Excel scope

The user-visible case

  • Whole-column references such as A:A.
  • Large rectangular ranges such as A1:XFD1048576.
  • Mostly blank sheets with a small number of defined cells.
  • Aggregation formulas that should run over the logical range, not allocate the dense range.

The engine problem

  • The formula result depends on the declared extent and defined entries.
  • Allocating every blank cell is wasteful and can hide semantic mistakes.
  • Replay must prove what was read without storing a dense shadow sheet.

Core Contract

SparseRangeReader is a worksheet-reference reader, not a rich object abstraction.

contract slice

Required operations

  • declared_extent: the logical range shape.
  • defined_cardinality: count of defined entries.
  • defined_iter: deterministic stream of defined entries.
  • read_at(coord): Defined(EvalValue) or Blank.
  • contains(coord): coordinate membership in extent.

Non-negotiables

  • Operations must not require O(extent) traversal.
  • Iteration order is contractual and replay-visible.
  • Blank and empty string must remain distinct.
  • UsedRange and formatting belong to sparse sheet structural management, not read_at value state.

What the reader must preserve

  • Never-assigned and assigned-then-cleared are both Blank at formula value level.
  • Empty string is Defined(EvalValue::Text("")).
  • COUNTBLANK, COUNTA, arithmetic coercion, and equality rely on this split.
  • Used range, formatting, comments, validation, and conditional formats may back sparse bounds, but they are not formula value state.

In Scope

W051 is deliberately narrow.

scope boundary

In W051

  • Worksheet cell/range semantics for whole-column and large-area reads.
  • Defined-entry versus blank behavior for formulas.
  • Aggregation and criteria-family functions where Excel behavior can be preserved sparsely.
  • Replay of declared extent, defined stream, iteration order, and result.

Out of W051

  • Generic virtualized arrays unrelated to worksheet references.
  • Queryable external objects and producer-specific pushdown protocols.
  • Generic rich producer/consumer protocol beyond the Excel sparse reader.
  • New non-Excel capability families.

First Function Slice

Start with functions that prove the sparse value semantics.

SUM COUNT COUNTA COUNTBLANK

Recommended first group

  • SUM: numeric aggregation and blank-as-zero behavior.
  • COUNT: numeric defined-entry counting.
  • COUNTA: defined nonblank counting, including empty string.
  • COUNTBLANK: blank counting over declared extent.

Later widening

  • AVERAGE, MIN, MAX after first semantics are stable.
  • Criteria-family functions after reference criteria handling is pinned.
  • Producer-side filtering only when replay records resolved iterators exactly.
The first group covers numeric, defined, blank, and empty-string behavior10 / 14

Replay

Replay must show what was read, not merely that the result matched.

evidence policy

Required artifact fields

  • Declared extent and coordinate system.
  • Defined-entry stream in deterministic order.
  • Blank and empty-string behavior evidence.
  • Function result and function input profile.
  • No-dense-materialisation check or counter.

Bad evidence

  • A result-only test over a small range.
  • A dense vector passed through a sparse-shaped API.
  • A replay artifact that cannot reconstruct the read stream.
  • A contract that hides iteration order as implementation detail.
W051 evidence needs traceability, not just correctness by accident11 / 14

Risks

The main failure modes are scope drift and fake sparsity.

controls

Watch for

  • Generic rich-data work sneaking into an Excel sparse range workset.
  • A dense implementation hidden behind sparse names.
  • Fixture-only backing stores reported as production substrate.
  • Result-only tests that do not prove iteration and replay.

Controls

  • Keep the scope bound to worksheet references.
  • Require operation-level complexity claims and checks.
  • Label fixture and production paths separately.
  • Make declared extent and defined stream mandatory artifacts.
Engineering quality bar: observable behavior and replay evidence13 / 14
