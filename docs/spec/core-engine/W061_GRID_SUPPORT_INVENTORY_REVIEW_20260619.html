Review stance

Executive Reading

The last few days produced substantial executable material. It is not only planning text: there are grid reference/profile types, provider implementations, a reference value machine, a scalar invalidation oracle, optimized storage candidates, spill ledgers, table overlays, hidden-row context, structural-edit machinery, seed corpus runs, and many scale/performance artifacts.

The main issue is not absence of substance. The issue is that the substance is concentrated in a few generated files and has crossed architectural boundaries that the docs themselves say must remain explicit. In particular, OxCalc now contains a large optimized R1C1 formula compiler/evaluator path. That may be useful as a characterization prototype, but it must not become a parallel formula engine competing with OxFml/OxFunc.

Primary risk Reference profile, grid kernel, function fast paths, invalidation, spill handling, scale harnesses, and tests are fused into mega-modules.
Primary opportunity The generated work has enough executable behavior to mine for real kernels, provider semantics, oracle tests, and performance constraints.
Recommended posture Freeze feature growth. Treat the current branch as an extraction source and evidence mine, not as the final architecture.
Decision target Classify each surface as contract, kernel, provider, oracle, optimized candidate, harness, evidence, or discard/prototype.
Important: the existence of retained artifacts and passing tests should not be interpreted as product completion. Many lanes are narrow exact slices, especially repeated-R1C1 optimized formulas and spill behavior. The evidence is valuable, but it is not a broad Excel grid claim.

Boundary doctrine

Intended Architecture

Target ownership flow

OxFml

Formula grammar, source spans, bind lifecycle, profile dispatch, normal-form envelopes, transform/render lifecycle.

OxFunc

Function semantics, argument preparation, value/reference coercion, host-info and reference provider traits.

OxCalc

strict-excel-grid profile semantics, grid state, provider implementation, dependency/invalidation, spills, visibility.

DnaTreeCalc

Non-grid proving profile. If the seam becomes awkward for tree references, the abstraction is too grid-shaped.

OxFml asks lifecycle questions. OxCalc answers grid questions. OxFunc mediates function/reference behavior.

Current generated shape

excel_grid_reference.rs

Profile payloads, bind profile, runtime provider, structured refs, text resolution, transforms, provider tests.

grid_reference_machine.rs

Reference machine, optimized sheet, fast-path formula compiler, provider, spill ledger, invalidation, structural edits, tests.

grid_runner.rs

Seed corpus definitions, runner, JSON artifact writer, invalidation reports, case tests.

grid_scale.rs

Scale profiles, performance assertions, procedural datasets, counters, retained run artifacts.

The concepts exist, but their boundaries are not yet product-grade.
Architectural invariant to protect: OxCalc may own the strict Excel grid profile and runtime provider. OxCalc should not grow a second formula engine. Formula parsing, binding, evaluator lifecycle, and function argument semantics need to remain anchored in OxFml/OxFunc public seams.

How to conduct the review

Recommended Review Process

Phase 1 - Freeze and classify

Step Output Acceptance test
Declare current generated grid branch an extraction source. Short status note in W061 or a review packet. No new formula families or spill features added until classification is done.
Classify public items from the four main files. Inventory table with category: contract, kernel, provider, profile, oracle, optimized, harness, evidence, discard. Every public type/function has an owner category and target module.
Classify tests. Test inventory: contract, characterization, regression, scale, prototype. Any code extraction names which tests must pass before and after movement.

Phase 2 - Extract stable foundations

Slice Proposed destination Definition of done
Bounds, addresses, rects, formula-cell identity grid_model Small module, no evaluator dependencies, focused unit tests.
Strict Excel reference payload and bind profile grid_profile Profile tests pass; no runtime storage types imported.
Runtime reference provider grid_provider Provider tests pass with both reference and optimized backing adapters.
Reference value oracle grid_reference_floor GridCalc-Ref tests pass; module remains mark-all-dirty and BTreeMap-based.
Invalidation oracle grid_invalidation_ref Dirty-closure tests pass; no dependency on optimized storage internals.

Phase 3 - Quarantine optimized candidate

Move optimized storage, repeated formula regions, plan cache counters, and scale profiles behind an explicit experimental or candidate module. The module can remain valuable, but its formula recognition must be reviewed as a temporary fast path. The target production path should consume OxFml-bound normal-form/reference packets and OxFunc function metadata.

Phase 4 - Matrix-driven rework

Structural edits, table lifecycle, spill repair, feature-rendered regions, and hidden-row provenance should be reviewed against explicit matrices. Do not let existing code define the matrix after the fact.

Phase 5 - Evidence curation

Retain the high-value seed and scale artifacts, but distill them into durable assertions. Debug runs with failing assertions should remain visible as probes, not proof.

// src/oxcalc-core/src/grid_reference_machine.rs
fn compile_r1c1_vlookup_expression(expression: &str) -> Option<GridOptimizedR1C1VLookupPlan> {
    let inner = expression.strip_prefix("VLOOKUP(")?.strip_suffix(')')?;
    let args = split_top_level_commas(inner)?;
    let [lookup_text, range_text, col_index_text, exact_mode_text] = args.as_slice() else {
        return None;
    };
    parse_r1c1_exact_lookup_mode(exact_mode_text)?;
    let (start, end) = parse_r1c1_range_or_single_reference(range_text)?;
    Some(GridOptimizedR1C1VLookupPlan {
        lookup: Box::new(parse_r1c1_scalar_expression(lookup_text)?),
