# W039: Core Formalization Release-Grade Successor Closure

## Purpose

W039 continues after the W038 non-promoting release-grade hardening tranche.

W038 improved the evidence floor, but it did not promote release-grade full verification, C5, pack-grade replay, Stage 2 policy, operated assurance services, fully independent evaluator diversity, broad OxFml display/publication closure, or general OxFunc kernels inside OxCalc. W039 is the successor lane that attacks those remaining blockers directly.

W039 remains a formalization, spec-evolution, and engine-improvement workset. It is not a test against a fixed initial spec. Current specs, TraceCalc behavior, optimized/core behavior, proof artifacts, TLA models, OxFml seam notes, and pack decisions are evidence surfaces that can force spec correction, implementation work, exact blocker carry-forward, or promotion only when the evidence warrants it.

The latest OxFml formatting intake is part of this scope: W073 now treats `VerificationConditionalFormattingRule.typed_rule` as the only metadata source for `colorScale`, `dataBar`, `iconSet`, `top`, `bottom`, `aboveAverage`, and `belowAverage`. The W072 bounded `thresholds` convention is intentionally ignored for those aggregate and visualization families, while `thresholds` remains valid only for scalar/operator/expression families where threshold text is the real rule input.

## Scope

### In Scope

1. Convert W038 residual successor lanes into exact W039 obligations with owners, evidence roots, promotion consequences, and spec-evolution hooks.
2. Resolve or retain exact optimized/core blockers for callable metadata projection, dynamic-reference counterparts, snapshot-policy counterparts, and capability-fence counterparts without match-promoting declared gaps.
3. Strengthen Lean and TLA evidence toward totality, proof/model coverage, and promotion predicates while preserving explicit assumptions and bounded-model limits.
4. Move Stage 2 evidence from bounded replay toward production partition-analyzer soundness and pack-grade replay equivalence where feasible.
5. Convert local alert/quarantine/history evidence toward operated continuous-assurance, retained-history, alert-dispatch, and cross-engine differential service artifacts.
6. Establish or explicitly block an independently implemented evaluator row set and cross-engine diversity surface.
7. Exercise OxFml seam breadth where OxCalc consumes it, including W073 typed conditional-formatting metadata, `format_delta`/`display_delta`, callable metadata, public consumer entry points, and the narrow `LET`/`LAMBDA` carrier boundary.
8. Reassess pack-grade replay, C5, and release-grade verification only after direct W039 evidence is bound.

### Out Of Scope

1. General OxFunc semantic-kernel formalization beyond the narrow `LET`/`LAMBDA` carrier seam consumed by OxCalc.
2. Direct edits to OxFml from this repo.
3. Promotion from proxy evidence, file-backed-only evidence, local simulation, declared gaps, or aggregate confidence.
4. Treating performance/scaling measurements as semantic correctness proof.
5. Freezing a production coordinator API from fixture-host or note-level packet convergence alone.

## Initial Guardrails

1. Release-grade claims require direct artifacts and gate-specific evidence.
2. TraceCalc remains the correctness oracle for covered reference behavior, not a substitute for optimized/core, service, or OxFml breadth evidence.
3. A declared gap is not a match.
4. Bounded proof/model artifacts are not total verification unless their assumptions and bounds are discharged for the claimed scope.
5. Stage 2 strategy changes require semantic-equivalence evidence that observable results are invariant for the promoted profiles.
6. Operated service claims require operated evidence, not local file-backed simulation.
7. Fully independent evaluator diversity requires independent implementation authority.
8. W073 aggregate and visualization conditional-formatting metadata is `typed_rule`-only for the named families; W072 threshold strings are not a fallback there.
9. `LET`/`LAMBDA` remains a narrow OxCalc/OxFml/OxFunc carrier seam, not general OxFunc kernel formalization inside OxCalc.
